# CI speed PR — highest-impact fixes from the 4-lane audit + build-tooling research

## Context

Main builds run 30-80 min. Measured root causes: (1) 13 images built **serially** in one dind pod, 45-52 min; (2) the 4 homelab images have **zero effective registry cache** (`--cache-from` refs never written by any `--cache-to`) — redlib fully recompiles `cargo build --release` every main build and churns digests → spurious redeploys; (3) that long job starves everything behind the 7.5 CPU/16Gi Kueue fence for up to 66 min; (4) two turbo config bugs kill remote-cache for cooklang/toolkit/tasknotes-server; (5) e2e (playwright install + sjer.red build) and resume (texlive) run on every PR regardless of affectedness. Research verdict: BuildKit stays the engine; **Docker Bake (GA)** is the orchestrator; end-state builder choice (shared buildkitd vs daemonless) is deferred to spikes — bake work carries over unchanged.

Operator decisions locked: one themed PR; auto-cancel stays; `release-please` regates on `verify`; request-shrinking must be cold-cache-safe; PR affected-scoping preserved; candidates push to ghcr directly (no local registry) in the future pod-smoke end state.

## Changes (this PR)

### 1. Parallel image builds with real caching — the 45-min fix

- **New root `docker-bake.hcl`**: 13 targets (9 app images: repo-root context + VERSION/GIT_SHA args; 4 homelab: `packages/homelab/images/<name>` contexts). Per-target registry `cache-from` AND `cache-to` (`mode=max,image-manifest=true`) — fixes the homelab zero-cache bug.
- **Rework `.buildkite/scripts/build-smoke-push-all.sh`**: one `docker buildx bake --builder ci --load` (parallel) replaces the serial loop; then serial smoke → tag → push per image. Smokes invoked directly (`bun run --cwd <pkg> smoke`) — turbo's `smoke`/`docker:build` are `cache:false` (`turbo.json:39-47`), nothing lost.
- **PR dry-run step (`pipeline.yml:256-289`)**: same bake path via a shared script, build+smoke only. Affected-scoping preserved: compute affected packages via turbo → pass only their bake targets, plus the 3 always-on families (documented turbo nested-affected bug). Guard fails loud on tool error.
- **Dockerfile layer fixes**: move birmel yt-dlp (`packages/birmel/Dockerfile:66-79`) and dpp codex install (`Dockerfile:115-116`) above `COPY . .`; delete dead per-package `.dockerignore` files (repo-root context → only root's applies).

### 2. Kueue queue starvation — the 66-min-wait fix

- **New `pod_light` anchor** in `pipeline.yml` (~lines 17-42): requests ~500m/1Gi, **limits unchanged** (cold-cache burst headroom). Applied to: build summary, npm publish, helm push, cooklang publish, deploy sites, tofu applies (github/infra/cloudflare), greptile gate, release step. `verify`/e2e/docker-e2e/images keep full-size pods.
- **`release-please` regate**: `depends_on: [images, sites]` → `verify` (`pipeline.yml:435`) — it consumes no image/site artifact; release-consuming steps keep their own gates.

### 3. Turbo remote-cache fixes

- New `packages/cooklang-for-obsidian/turbo.json`: `build.outputs: ["main.js"]` (esbuild outfile is pkg-root `main.js`, not `dist/**` — task currently caches nothing; source of the "no output files found" warning).
- Root `turbo.json`: `remoteCache.uploadTimeout: 300` — the 180 MB `toolkit`/`tasknotes-server` compiled binaries exceed the default timeout → remote-miss every build.

### 4. PR scoping for the two unconditional heavy steps

- `e2e` (`pipeline.yml:124`) and `resume-build` (`:160`): on non-main branches, check affectedness via turbo (`sjer.red` / `@shepherdjerred/resume`) and exit 0 with an explicit "unaffected — skipping" message; any tool error fails the step. Main always runs both (e2e's dist artifact feeds `sites`).

### 5. Per-pod setup tax (~45-155s × ~11 pods per build)

- **Restore the git-mirror PVC + checkout config** in `packages/homelab/src/cdk8s/src/resources/argo-applications/buildkite.ts`: re-add the `buildkite-git-mirrors` `KubePersistentVolumeClaim` (RWX, `NVME_STORAGE_CLASS`) and `config.default-checkout-params.gitMirrors` (volume + `lockTimeout: 300`) exactly as removed in 4f08817be. The removal's stated rationale ("replatformed CI does a plain shallow checkout") is stale — the pipeline does **full** `--no-tags` clones (turbo merge-base), measured 52s cold per pod on every main build vs 8-10s mirrored. New comment states the real rationale + measurements. Takes effect on ArgoCD sync post-merge; benefits every subsequent build.
- **Bake a warm bun cache into ci-base** (`.buildkite/ci-image/Dockerfile` + `.buildkite/scripts/build-ci-image.sh`): during image build, run a workspace `bun install --frozen-lockfile` against the repo context with `BUN_INSTALL_CACHE_DIR` set to a fixed path, then discard `node_modules`/workspace — keep only the package cache layer. Add `bun.lock` to the image content-hash key (currently `.mise.toml` only) so the cache layer rebuilds on dep changes; a stale cache degrades gracefully (bun fetches missing packages) rather than failing. Measured today: installs range 13-72s on cache warmth; this pins ~13s.

## Explicitly deferred (ranked follow-ups, not this PR)

1. **End-state builder + pod-smokes**: build → push `:sha` candidates to ghcr → k8s smoke pods (same containerd as prod, no privilege, no `--load`) → promote `:latest`+digest commit-back post-smoke. Two bounded spikes decide the builder: single-image shared-buildkitd trial (ZFS PVC + snapshots) vs staying daemonless. Bake file transfers either way.
2. Scoped retry-on-exit-1 for install-flake-prone steps.

## Expected outcome

Images step 45-52 min → ~10-15 min (parallel + redlib/homelab cache hits from build #2); light steps admit alongside images instead of queueing 3-66 min; release lane starts ~40 min earlier; checkout 52s → ~10s and `bun install` pinned ~13s across ~11-15 pods/build (~45-90s off the critical path, fast-path variance collapses); cooklang/toolkit remote-cache functional; PRs skip playwright/texlive when unaffected. Steady-state main ~10-12 min, PRs ~4-6 min.

## Implementation deltas (found while building)

1. **The version commit-back was a dead stub.** `bun scripts/update-versions.ts`
   (bare, as the pipeline called it) hits `usage()` → exit 1; no digest plumbing
   from image pushes ever existed in the replatform, and the release step has
   never passed on this pipeline (checked builds 5641-5663). Fix shipped in this
   PR: `bake-images.sh --push` records pushed digests to build meta-data
   (`image-digests` JSON), a new main-only `:package: version commit-back` step
   (gated on `images`) feeds them to `update-versions.ts --commit-back
"2.0.0-$BUILD_NUMBER" --digests …`, and `rewriteVersionsFile` now skips
   digest-unchanged entries (so a no-change merge redeploys nothing — this also
   makes the homelab-cache fix the thing that stops their every-merge digest
   churn). release-please itself regated on `verify` as planned.
2. **ci-image has no content-hash tag key to extend.** The Dockerfile comment
   claimed the tag is the .mise.toml content hash, but `build-ci-image.sh` tags
   `:sha`/`:latest` and rebuilds every main build via registry cache. The warm
   bun cache therefore needs no key change — its layer invalidates when the
   COPY'd manifests/lockfile change, which is exactly the desired cadence.
   (Stale Dockerfile comment corrected in the same edit.)
3. **`is-affected.ts` (new, `scripts/`)** backs the e2e/resume PR gates:
   prints `affected`/`unaffected` only, fails loudly on turbo errors AND on
   nonexistent package names (a rename can't silently disable a gate).
