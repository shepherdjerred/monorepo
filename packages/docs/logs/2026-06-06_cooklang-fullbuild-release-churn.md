# Cooklang Manifest-Bump Churn on Full Builds

## Status

Complete ŌĆö shipped in [PR #1038](https://github.com/shepherdjerred/monorepo/pull/1038)
(branch `claude/intelligent-buck-86ba99`), awaiting CI/merge.

## Summary

`chore(cooklang): bump plugin manifest version` PRs kept landing even though no
cooklang source files changed. The [2026-05-25 fix](2026-05-25_cooklang-release-loop.md)
stopped the _tight_ self-perpetuating loop (bump-merge immediately re-publishing),
but a second, independent trigger remained: **every full/infra build cut a new
cooklang release.** Bumps continued v1.0.13 ŌåÆ v1.0.19 after that fix, now
hours/days apart (driven by unrelated builds) instead of minutes apart.

## Root cause

Two compounding issues in the CI pipeline generator:

1. **Over-trigger (`scripts/ci/src`).** `fullBuildResult()` hard-coded
   `cooklangChanged: true`, and the release gate in `pipeline-builder.ts:288`
   fired on `affected.buildAll || affected.cooklangChanged`. Any change to
   `.dagger/`, `scripts/ci/`, `.buildkite/`, `bun.lock`, `package.json`, or
   `tsconfig*.json` triggers a full build ŌåÆ cooklang release, regardless of
   whether cooklang changed. The v1.0.18 + v1.0.19 pair (identical timestamps,
   2026-06-06) were two concurrent full builds ŌĆö from a batch of `.dagger`/`scripts/ci`
   commits ŌĆö each cutting a release.

2. **Unconditional bump (`.dagger/src/release.ts`, `cooklangPublishHelper`).**
   `new="$major.$minor.$((patch + 1))"` always bumps, then `gh release create`
   fires for the new tag (which never pre-exists). The in-helper
   `git diff --cached --quiet` idempotency guard (release.ts:802) is dead code:
   the manifest `version` is rewritten _before_ the diff check, so the diff is
   never empty. So the publish always emits a new version ŌåÆ commit-back opens a PR.

## Fix (this session ŌĆö issue 1)

Decouple "rebuild cooklang" (every full build, for safety) from "publish
cooklang" (only on real source change).

- `scripts/ci/src/change-detection.ts`
  - New `hasCooklangSourceChange(changedFiles)` ŌĆö true only for files under a
    cooklang package that are NOT the auto-bump artifacts (`manifest.json` /
    `versions.json`). Excluding the artifacts also defuses the interleaved-bump
    diff-window edge case.
  - `fullBuildResult(cooklangChanged)` and `buildScopedResult(..., cooklangChanged, ...)`
    now take the flag instead of hard-coding / package-set-deriving it.
  - `detectChanges` computes `cooklangSourceChanged` once from the diff and
    threads it to every result. Forced full builds (`[full-build]` / `FULL_BUILD`)
    pass `false` ŌĆö they rebuild everything but never publish cooklang.
- `scripts/ci/src/pipeline-builder.ts:288` ŌĆö gate is now
  `releaseBuild && !affected.isAutoGenerated && affected.cooklangChanged`
  (dropped `affected.buildAll ||`).

Net behavior: infra/full builds rebuild & retest cooklang but publish nothing;
real cooklang source/dep changes (incl. a Renovate `chevrotain` bump that touches
`packages/cooklang-for-obsidian/package.json`) still cut a release.
