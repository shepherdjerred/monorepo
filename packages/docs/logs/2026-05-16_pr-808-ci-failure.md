# PR 808 CI Failure

## Context

Question: why did CI fail for `shepherdjerred/monorepo#808`?

PR #808 only changes `.buildkite/ci-image/VERSION` from `406` to `407`.

## Findings

Buildkite build `2435` failed on `buildkite/monorepo/pr/no-entry-sign-ci-base-version-guard`.

The failed job was `:no_entry_sign: CI Base Version Guard` with exit status `1`. Its command was:

```bash
echo '.buildkite/ci-image/VERSION is updated by CI after ghcr.io/shepherdjerred/ci-base is published. Revert this file from the PR.' && exit 1
```

This guard is generated by `scripts/ci/src/pipeline-builder.ts` when change detection sees `.buildkite/ci-image/VERSION` changed in a non-generated commit. The intended flow is that CI publishes `ghcr.io/shepherdjerred/ci-base` first, then CI's commit-back step updates `.buildkite/ci-image/VERSION`; normal PRs should not bump that file directly.

The commit was auto-generated: commit `943fbc11f1fce0e4d37e031553bc7baebb34db41` was authored by `CI Bot` and has commit message body `Auto-Generated: ci-bot`. The reason the guard still fired is likely that `scripts/ci/src/change-detection.ts` detects bot commits from `BUILDKITE_MESSAGE`, while Buildkite build `2435` exposed only the subject (`chore: bump ci-base image to 407`) in the build message. The trailer was therefore not visible to `isAutoGeneratedCommit()`, so the generated PR was misclassified as a normal PR.

Ordering is correct for real ci-base image source changes:

1. `ciBaseImageBuildStep()` builds the image after `quality-gate`.
2. `ciBaseImagePushStep()` is `main`-only and depends on `build-ci-base`; it tags `ghcr.io/shepherdjerred/ci-base:<next-version>` and `latest`.
3. `ciBaseVersionCommitBackStep()` is `main`-only and depends on `push-ci-base`; it opens/updates `chore/ci-base-version-bump-pending` only after the image push step succeeds.

A robust fix must handle both builds generated by the auto-created PR:

- PR build on `chore/ci-base-version-bump-pending`: read the full commit message from `git log -1 --pretty=%B` so the `Auto-Generated: ci-bot` trailer is visible and the VERSION-only PR becomes a no-op pipeline.
- Post-merge `main` build: because `gh pr merge --auto --merge` creates a merge commit, the HEAD commit may not have the trailer. The detector should also recognize the merge message/branch for `chore/ci-base-version-bump-pending`, or add an explicit ci-base VERSION commit-back fast path. Otherwise the PR build can be fixed while the merge build still trips the same guard.
