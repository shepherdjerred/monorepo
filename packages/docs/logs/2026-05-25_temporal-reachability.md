# Temporal Reachability Check

## Status

Complete

## Summary

Temporal was unreachable from the local default address and from the cluster service because the Temporal server pod was crash-looping.

Evidence gathered on 2026-05-25:

- `nc -vz 127.0.0.1 7233` returned `Connection refused`; no local Temporal dev server or port-forward was listening.
- `temporal operator cluster health --address 127.0.0.1:7233` also failed with `connection refused`.
- In namespace `temporal`, `temporal-temporal-server-54d8886cd4-6wt8z` was `CrashLoopBackOff`, so `temporal-temporal-server-service` had no ready v1 Endpoints.
- The Temporal server log showed `pq: password authentication failed for user "temporal"` and then `Unable to setup SQL schema`.
- The current `Postgresql` CR has `pg_hba` entries requiring `scram-sha-256` for the `temporal` and `temporal_visibility` databases.
- Recent homelab commits changed Temporal Postgres auth from `md5` to `scram-sha-256` and added `password_encryption: "scram-sha-256"`.

The likely root cause is an auth-method migration mismatch: `pg_hba` now requires SCRAM, but existing role password hashes are not automatically rehashed when `password_encryption` changes. Until the role password is rotated or the auth method is rolled back, the Temporal server cannot authenticate to Postgres and cannot open port 7233.
