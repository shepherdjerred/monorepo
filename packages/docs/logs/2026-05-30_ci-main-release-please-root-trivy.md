# CI red on main — release-please root guard + Trivy CVEs

## Status

Complete (pending CI verification on the PR)

## Symptom

`main` CI was red. Build [#3068](https://buildkite.com/sjerred/monorepo/builds/3068)
(commit `db016cf20`) had two failed jobs:

- `:bookmark: Release Please` — exit 1 (the real blocker)
- `:shield: Trivy Scan` — exit 1 (HIGH CVEs in lockfiles)

## Root causes

### 1. Release Please — Claude root guard

The auto-refine step added 2026-05-29 in `228c9c948` chains three commands with `&&`:

```
release-please release-pr …        ✅
claude -p "$REFINE_PROMPT" … --dangerously-skip-permissions …   ❌
release-please github-release …    (never reached)
```

The `releasePleaseHelper` container (`.dagger/src/release.ts`) is
`dag.container().from(BUN_IMAGE)` with **no `.withUser()`**, so it runs as
**root**. Recent Claude Code refuses the flag under uid 0:

```
--dangerously-skip-permissions cannot be used with root/sudo privileges for security reasons
```

Verified the exact guard in the installed Claude binary (2.1.157):

```js
if (
  process.getuid() === 0 &&
  process.env.IS_SANDBOX !== "1" &&
  !CLAUDE_CODE_BUBBLEWRAP
)
  (console.error(
    "--dangerously-skip-permissions cannot be used with root/sudo privileges...",
  ),
    process.exit(1));
```

The feature had essentially never passed since it landed.

### 2. Trivy — HIGH CVEs in three lockfiles

- `packages/better-skill-capped` — axios 1.15.2 → CVE-2026-44492, CVE-2026-44494 (fixed 1.16.0)
- `packages/birmel` — axios 1.15.2 (override-only, transitive) → same two CVEs
- `packages/discord-plays-pokemon` — tmp 0.2.5 (transitive via selenium-webdriver) → CVE-2026-44705 (fixed 0.2.6)

## Fix

- `.dagger/src/release.ts`: added `.withEnvVariable("IS_SANDBOX", "1")` to the
  release-please container. This is Claude Code's documented escape hatch for
  trusted root automation containers. Switching to a non-root user was rejected
  because the in-container `> /usr/local/bin/git-askpass` write and the
  root-owned `/workspace` git operations depend on root. The container is
  ephemeral and runs a fixed, code-reviewed prompt scoped only by the GitHub
  App token — exactly the sandbox case the flag targets.
- `packages/birmel/package.json`: override axios 1.15.2 → 1.16.0
- `packages/better-skill-capped/package.json`: axios `^1.15.2` → `^1.16.0` (resolves 1.16.1)
- `packages/discord-plays-pokemon/package.json`: added `tmp: 0.2.6` override
- Regenerated all three `bun.lock` files; verified resolved versions
  (axios@1.16.0, axios@1.16.1, tmp@0.2.6).

## Verification

- `bun scripts/check-dagger-hygiene.ts` → No violations
- `better-skill-capped` `tsc --noEmit` → clean (direct axios consumer)
- birmel/dpp consume bumped deps only transitively (override-only) → no app-code risk
- `.dagger` local `tsc` can't run (the `@dagger.io/dagger` SDK module is
  generated by the Dagger engine in CI, absent in a plain checkout); the change
  is a single `.withEnvVariable(...)` call identical in form to the adjacent
  `BUN_INSTALL` line, so it is type-sound. Dagger typecheck runs in CI.
