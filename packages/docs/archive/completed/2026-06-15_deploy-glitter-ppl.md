# Deploy glitter.html to ppl.glitter-boys.com

## Status

Complete — shipped in PR #1257; bucket + DNS live, HTML version-controlled with CI auto-deploy.

The plan originally called for one-shot manual upload. Mid-implementation the user asked to version-control the HTML under a new `packages/glitter/` package so it deploys via CI — that addendum is captured in the Session Log below.

## Context

The user has a self-contained D3 social-graph visualization at `~/Downloads/glitter.html` (8.9 KB, "Glitter Graph" — the friend-group network). They want it served publicly under their existing Cloudflare-managed `glitter-boys.com` zone, on the new subdomain `ppl.glitter-boys.com`. Deploy mode is **one-shot manual upload** of the HTML — but the supporting infra (bucket, DNS, Caddy route) still needs PR'd code changes because all of it is GitOps-managed.

The existing `glitter-boys.com` zone (managed by OpenTofu) already has DNSSEC, CAA, TLS hardening, and Fly.io CNAMEs for `beta.` and `prod.` We are adding a brand-new subdomain that lands on the homelab's standard SeaweedFS-backed static-site stack — the same plumbing that serves `clauderon.com`, `ts-mc.net`, `better-skill-capped.com`, etc.

The HTML loads `d3.v7.min.js` from the official `d3js.org` CDN. The homelab's default response headers do **not** set a CSP, so the external D3 will load fine without any override.

## Approach

Use the existing static-site abstraction unmodified — one bucket, one DNS record, one entry in `sites.ts`, one `aws s3 cp`. No new patterns.

### Naming

- **Bucket**: `glitter-boys-ppl` (namespaced under the parent zone to leave room for future apex/`graph`./etc. buckets without ambiguity).
- **Hostname**: `ppl.glitter-boys.com`.
- **Object key**: `index.html` (matches the static-site abstraction's default `indexFile`).

### 1. `packages/homelab/src/tofu/seaweedfs/buckets.tf`

Append next to the other static-site buckets (e.g. after `cook`, line ~36):

```hcl
resource "aws_s3_bucket" "glitter_boys_ppl" {
  bucket = "glitter-boys-ppl"
}
```

### 2. `packages/homelab/src/tofu/cloudflare/glitter-boys-com.tf`

Append a new record (the existing `beta`/`prod` records stay untouched — they belong to Fly.io):

```hcl
# Homelab static site (Cloudflare Tunnel → s3-static-sites Caddy → glitter-boys-ppl bucket)
resource "cloudflare_dns_record" "glitter_boys_com_cname_ppl" {
  zone_id = cloudflare_zone.glitter_boys_com.id
  ttl     = 1
  name    = "ppl"
  type    = "CNAME"
  content = "3cbdc9a6-9e79-412d-8fe1-60117fecd4d3.cfargotunnel.com"
  proxied = true
}
```

Tunnel ID is the shared homelab tunnel already used by every `*.sjer.red` site, `better-skill-capped.com`, etc. (`grep cfargotunnel packages/homelab/src/tofu/cloudflare/sjer-red.tf`).

### 3. `packages/homelab/src/cdk8s/src/resources/s3-static-sites/sites.ts`

Append one line to the `staticSites` array (matches the minimal pattern used by `clauderon.com`, `ts-mc.net`, `better-skill-capped.com`):

```ts
{ hostname: "ppl.glitter-boys.com", bucket: "glitter-boys-ppl" },
```

Defaults take care of everything else: index.html, default response headers (HSTS, X-Frame-Options DENY, nosniff, Referrer-Policy, Permissions-Policy), and a `TunnelBinding` generated by `createCloudflareTunnelBinding` via `disableDnsUpdates: true` (because DNS is OpenTofu-managed).

No `responseHeaders` override needed — the default header set doesn't include CSP, so the external `d3js.org` script loads unblocked.

### 4. Upload the HTML

After the bucket exists (step 1 applied), upload from the laptop:

```bash
op run --env-file=.env -- aws s3 cp \
  ~/Downloads/glitter.html \
  s3://glitter-boys-ppl/index.html \
  --endpoint-url https://seaweedfs.sjer.red \
  --content-type "text/html; charset=utf-8"
```

(Use whatever AWS profile/op envelope already works for the user's other SeaweedFS `aws s3 cp` calls — same creds used by `toolkit pr asset`'s `seaweedfs` profile.)

## Apply / merge order

1. **Commit** the three file changes on `feature/glitter-ppl` and open a PR.
2. **Apply OpenTofu** locally (or once PR'd, per the user's normal flow):
   - `op run --env-file=.env -- tofu -chdir=src/tofu/seaweedfs plan && … apply` — creates the bucket.
   - `op run --env-file=.env -- tofu -chdir=src/tofu/cloudflare plan && … apply` — creates the DNS record.
3. **Merge the PR** — ArgoCD syncs the new Caddy site config + `TunnelBinding`, so the Cloudflare Tunnel now routes `ppl.glitter-boys.com` to the s3-static-sites Caddy service, which proxies to `s3://glitter-boys-ppl/`.
4. **Upload the HTML** (step 4 above).

## Notes / caveats

- **HSTS already covers the new subdomain.** The zone's `security_header` setting has `include_subdomains = true` and `max_age = 86400` — so anyone whose browser has visited `glitter-boys.com` will require HTTPS for `ppl.` too. With `proxied = true`, CF edge serves HTTPS natively, so this is fine.
- **No interaction with the Fly.io `beta.`/`prod.` records.** Those keep pointing at `*.fly.dev` (unproxied) and are unaffected.
- **One-shot upload means manual updates.** Re-uploading the HTML in the future is another `aws s3 cp`. If the user later wants edits versioned, the migration is straightforward: add a `packages/glitter-boys/` source dir, register it in `scripts/ci/src/catalog.ts` `DEPLOY_SITES` and `PACKAGE_TO_SITE`, and CI auto-syncs the bucket on merge (the `better-skill-capped` pattern at `catalog.ts:210-217`).

### Aws-chunked content-encoding gotcha

The first manual upload (`aws s3 cp`) persisted `Content-Encoding: aws-chunked` on the object — that's the AWS CLI v2 default checksum/chunked-signing transport leaking into object metadata via SeaweedFS. Browsers don't recognize `aws-chunked` and download instead of render. Fix: prefix the call with `AWS_REQUEST_CHECKSUM_CALCULATION=when_required`. The object now has no `ContentEncoding`, just `text/html`. (The CI deploy path uses `aws s3 sync` which doesn't hit this — only `put-object`/`cp` do.)
