# Plan: `toolkit deployed` — "is my commit deployed?"

## Context

Tracing a commit through the homelab deploy pipeline (build → version bump →
`versions.ts` → ArgoCD sync → running pod) was a manual, multi-step chore
documented in [guides/2026-04-06_is-commit-deployed.md](../guides/2026-04-06_is-commit-deployed.md).
`toolkit deployed [<selector>]` automates it end to end, answering "is my
change/service deployed to the homelab k8s?" in one call. Scope: homelab/K8s
services (static sites + npm are out of v1); data sources auto-degrade
git → GitHub → live cluster.

## How deployment actually works — TWO builds, not one

1. **Build 1** (feature merge): builds+pushes `ghcr.io/…/<svc>:2.0.0-A@sha256:…`,
   runs version-commit-back → opens the auto-merge `chore: bump image versions to
2.0.0-A` PR. Its ArgoCD sync runs against the _old_ `versions.ts`, so the new
   image is **not live yet**.
2. **Build 2** (the auto-generated bump commit B lands on main): image rebuilds
   skipped (`versionBumpOnly`/`isAutoGenerated`), but `versions.ts` is under
   `packages/homelab` ⇒ `homelabChanged` ⇒ cdk8s synth → helm push → ArgoCD sync.
   **This is the build that deploys.** (Confirmed in `scripts/ci/src/pipeline-builder.ts`.)

A commit is deployed when: it's an **ancestor** of the bump that **wrote the
affected service's current digest**, ArgoCD's chart `2.0.0-<build>` ≥ that bump,
and the running pod's `imageID` digest == the pin. The pod-digest match is the
gold signal.

## Gotchas (each verified by live trace, encoded in the code)

| #   | Trap                                                                                                                             | Tool rule                                                                                             |
| --- | -------------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------- |
| 1   | Linear `git log` order ≠ ancestry (bumps cut on side branches). `c60a7a0e4` sits below bump 3637 but isn't in it → not deployed. | `git merge-base --is-ancestor`, never log order.                                                      |
| 2   | Bump membership ≠ service rebuilt.                                                                                               | Pickaxe the **digest** (`git log -S<digest>`) to find the writing bump; ancestry-test against it.     |
| 3   | Promotion: scout beta=`3637`, prod=`2985`; prod's digest was _written_ by build `3016`, not 2985.                                | Never parse the tag number for ancestry; use the writing bump. prod PENDING says "manually promoted". |
| 4   | `argocd app get` needs `--grpc-web`; its revision is chart `2.0.0-<build>`, not a SHA.                                           | Pass the flag; compare build numbers numerically.                                                     |
| 5   | Pod digest is in `imageID` (`<repo>@sha256:…`), and `.image` may be a bare config sha.                                           | Match on both `image` and `imageID`.                                                                  |
| 6   | Multi-variant products (scout, starlight) each have their own ArgoCD app.                                                        | Per-variant registry → own argoApp/pods.                                                              |

## Command shape

```
toolkit deployed [<selector>] [--commit <ref>] [--json] [--no-cluster] [--no-github]
```

Selector resolves against the service registry first, else a git ref:
`(none)`/`<commit>` → commit-centric (auto-detect affected services);
`<service>` → all variants, commit defaults to the latest commit for that
package; `<service>/<variant>` (or `scout-prod`) → one product.

Verdict ladder: `NOT_MERGED → PENDING → NO_IMAGE → PINNED → SYNCED → RUNNING`.
Exits non-zero unless every affected variant is RUNNING.

## Files (all under `packages/toolkit/`)

- `src/handlers/deployed.ts` — arg parse, routes to command.
- `src/commands/deployed/deployed.ts` — orchestrate layers, verdict, format.
- `src/lib/deployed/git.ts` — repo-root detect, ancestry, changed packages,
  `:/`-rooted pathspecs (cwd-independent), digest pickaxe.
- `src/lib/deployed/versions-file.ts` — regex parser for first-party pins.
- `src/lib/deployed/catalog.ts` — service registry + selector resolution.
- `src/lib/deployed/argocd.ts` — `argocd app get <app> --grpc-web -o json`.
- `src/lib/deployed/kubectl.ts` — cluster-wide pod scan, digest match.
- `src/lib/deployed/github.ts` — PR / open-bump-PR lookup via `gh`.
- `src/lib/deployed/{types,schemas,format}.ts`.
- `test/deployed/{catalog,versions-file}.test.ts` — drift guard (every registry
  versionKey exists in live `versions.ts`) + parser tests.
- Wiring: `src/index.ts` case + usage; `package.json` `test:unit` includes
  `test/deployed`.
