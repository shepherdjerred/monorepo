# Streambot: Subtitle Track-Picker Select Menu

## Context

`/stream subtitles` currently takes free-text `mode` (on/off) and `language` (e.g. `en`, `en.forced`) options — the user has to guess a language code and hope it matches what's actually available. The bot already has rich subtitle-track discovery/ranking logic (sidecar file parsing, embedded ffprobe streams) used internally to auto-pick the best match; this plan surfaces that as a Discord select menu so the user picks from the _actual_ available tracks for the currently-playing video instead of guessing.

Verified via a real `yt-dlp --dump-single-json` call against a live YouTube URL: the JSON already contains `subtitles`/`automatic_captions` dicts (`{ [langCode]: [{ext, url, name}, ...] }`) that the bot's `YtdlpInfoSchema` currently drops — no second kind of yt-dlp call is needed to _enumerate_ what's available, just a schema extension. The `url` fields in that dict are signed and time-limited (`expire=` param), so they must never be persisted or fetched directly — the actual download at apply-time re-invokes yt-dlp fresh with a pinned language, exactly like today's flow already does.

This was designed with an Explore pass (verified all citations against the live tree, confirmed the yt-dlp JSON shape with a real invocation) and a Plan-agent design review that surfaced several real bugs in the draft (missing `defer()`, a `strictObject` schema-extension trap for resume-state persistence, a double-invocation race, and the `awaitMessageComponent` vs `createMessageComponentCollector` API fork) — all resolved below.

## Design

### 1. Model a specific track, not just a language (`src/sources/source.ts`)

Added a `SubtitleTrackRef` discriminated union and threaded it onto `SubtitlePref` as `trackRef?`. This is a **precise pick** (an exact sidecar file / embedded stream index+codec / yt-dlp language+auto-flag), distinct from the existing `language` field (a preference used for ranking).

```ts
export const SubtitleTrackRefSchema = z.discriminatedUnion("kind", [
  z.strictObject({ kind: z.literal("off") }),
  z.strictObject({ kind: z.literal("sidecar"), file: z.string().min(1) }),
  z.strictObject({
    kind: z.literal("embedded"),
    subtitleIndex: z.number().int().nonnegative(),
    codec: z.string().min(1),
  }),
  z.strictObject({
    kind: z.literal("ytdlp"),
    lang: z.string().min(1),
    autoGenerated: z.boolean(),
  }),
]);
```

Added `trackRef: SubtitleTrackRefSchema.optional()` to the existing `SubtitlePrefSchema` — edited the Zod schema itself (not just a TS type), since `SubtitlePrefSchema` is `z.strictObject` and embedded inside `PersistedQueuedSchema`/`PersistedCurrentSchema` (`src/state/persistence.ts`), which round-trip through `safeParse` on every resume-state load. No `url`/signed data is ever stored in `trackRef`, so a persisted pick surviving a bot restart is safe and meaningful.

### 2. Enumerate candidates without picking

- `subtitles.ts`: extended the `SubtitleCandidate` union with a third variant: `{ kind: "ytdlp"; lang: string; name: string | null; autoGenerated: boolean; modifier: null }`. Updated `sourceScore`/`candidateTieBreak` to order it after sidecar/embedded (never actually mixed with them in practice — a source is either a file or a URL/search, never both). Added a pure mapper `listYtdlpSubtitleCandidates(subtitles, automaticCaptions, includeAuto): SubtitleCandidate[]` — one candidate per language key per manual/auto dict.
- `ytdlp.ts`: extended `YtdlpInfoSchema` with `subtitles`/`automatic_captions` as `z.record(z.string(), z.array(z.object({ name: z.string().optional() }))).optional()` — deliberately omitting `ext`/`url` so the signed URL is never even parsed. Added `buildSubtitleEnumerationArgs` (same as `buildInfoArgs` minus `-f best`, meaningless for an enumeration-only probe) and `listSubtitleTracksForYtdlp(config, source, signal): Promise<SubtitleCandidate[]>` — a fresh `--dump-single-json` call, separate from the one already made at play-time resolution (confirmed the right call over threading the manifest through `PlaybackContext`: the download call still has to re-run regardless since enumeration URLs are signed/expiring, and the command is already a deferred, human-latency-bound interaction).
- `subtitle-io.ts`: added `listSubtitleCandidatesForFile(config, filePath, signal): Promise<SubtitleCandidate[]>` wrapping the existing (private) `gatherSidecarCandidates`/`probeEmbeddedCandidates` + `rankSubtitleCandidates`.
- New `src/sources/subtitle-candidates.ts`: `listSubtitleCandidatesForSource(config, source, signal)` dispatcher switching on `source.kind`.

### 3. Apply an exact pick, bypassing ranking (`subtitle-io.ts`)

Added `resolveExactTrackForFile`/`resolveExactTrackForYtdlp` helpers, invoked at the top of `resolveSubtitleForFile`/`resolveSubtitleForYtdlp` when `pref?.trackRef` is present:

- `{kind:"off"}` → `undefined` immediately.
- `{kind:"sidecar", file}` → `stageSidecar` directly, no gather/rank.
- `{kind:"embedded", subtitleIndex, codec}` → `extractEmbeddedTrack` directly, no re-probe (the codec travels in the trackRef so the cache key — which only hashes path+size+mtime+index — stays valid).
- `{kind:"ytdlp", lang, autoGenerated}` → the existing `ytdlpSubtitleArgs` builder, unmodified, just pinned to a single language + auto-flag instead of the config-driven list. Refactored the shared download/pick/clean tail into `downloadYtdlpSubtitle()` so both the ranking path and the exact-pick path share one implementation.
- A failed apply degrades to `undefined` exactly like the existing auto-pick path — no new error surfacing.
- An invariant violation (a `ytdlp` trackRef reaching the file resolver, or vice versa) throws loudly rather than silently falling back — this can only happen from a bug in the caller, never from real user input.

### 4. Discord select-menu UI (`src/discord/subtitle-menu.ts`, new)

- Pure `encodeTrackRef`/`decodeTrackRef`: `trackRef ↔` opaque string value (`off`, `sidecar:<file>`, `embedded:<idx>:<codec>`, `ytdlp:<lang>:<auto|manual>`), validated against `SubtitleTrackRefSchema.parse()` on decode.
- `sendSubtitleMenu(interaction, candidates): Promise<string | null>` — presents a `StringSelectMenuBuilder` with an "Off" option plus up to 24 ranked candidates (25 discord.js hard cap; truncates by the caller's ranked order with a visible "(+N more not shown)" note). Uses `message.awaitMessageComponent({ filter, time: 2min })` — the single-pick primitive, not `pagination.ts`'s persistent-collector pattern (that one exists for pagination's multi-click, self-contained re-render UX). Any rejection (timeout or otherwise) resolves as `null`; components are cleared on every exit path.
- `CommandInteraction` gained `replySelectMenu`, implemented in `command-bot.ts`'s `adapt()` alongside the existing `replyPaginated` wiring.

### 5. Command flow — menu-only, no free-text fallback

`/stream subtitles` dropped its `mode`/`language` string options entirely (they'd only existed for one prior session, unmerged) — every invocation now shows the picker.

- `SessionHandle` gained `listSubtitleCandidates`, `hasPendingSubtitleMenu`, `claimSubtitleMenu`, `releaseSubtitleMenu` — wired in `session-manager.ts`'s `handleFor`, reading `session.actor.getSnapshot().context.current?.source` (the layer with access to the raw `Source`; `command-handler.ts` only ever sees a projected `PlaybackView`).
- `handleSubtitles` rewrite: existing "nothing playing"/permission checks → single-flight claim (reject with an ephemeral reply if already open) → `defer()` → `listSubtitleCandidates` (empty → immediate "No subtitle tracks were found" reply, skip the menu round-trip) → `replySelectMenu` (`null` → "Selection timed out.") → decode pick → dispatch `CHANGE_SUBTITLES` (event shape unchanged — the machine already forwards whatever `SubtitlePref` it receives) → generic confirmation. `releaseSubtitleMenu()` runs in a `finally` so the guard always clears.
- Single-flight guard: `pendingSubtitleMenu: boolean` on the in-memory `Session` record.
- Extracted `chaptersText`/`nowPlayingText`/`queueText`/`buildSubtitlePref`/`subtitlesSuffix` out of `command-handler.ts` in the prior session's refactor already left room for this; no further file-size refactor was needed here.

### Caveats

- This plan builds directly on top of the earlier, same-session "list pagination / tips / subtitle-restart / play-error-fix" work on the same branch — the `/stream subtitles mode:on|off [language]` command that plan shipped is what this plan replaces entirely. Since neither has been merged/deployed yet, this was a clean in-place replacement with no back-compat shim.
- The yt-dlp enumeration call (`listSubtitleTracksForYtdlp`) adds real latency to `/stream subtitles` for URL/search sources (a full `--dump-single-json` metadata probe) — acceptable per the design (deferred, human-latency-bound interaction), but worth knowing if a future session investigates the command feeling slow for yt-dlp sources specifically.
- `subtitle-io.ts` and `subtitles.ts` are getting large with three source kinds now sharing ranking machinery; if a fourth subtitle source type is ever added, revisit whether `SubtitleCandidate`'s single shared-shape union (with `lang`/`modifier` fields that don't really apply to `ytdlp`) is still the right model, or whether per-kind ranking should split out.
