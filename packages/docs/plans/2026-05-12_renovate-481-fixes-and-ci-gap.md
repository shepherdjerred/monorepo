# Unbreak `main` and close the renovate-481 CI gap

## Status

In Progress

## Context

PR #773 ("renovate dashboard 481") merged on 2026-05-12 with a green Buildkite status, but main has been red ever since (builds #2332 and #2336). Investigation revealed two distinct problems:

1. **Four jobs broke on main** that the PR could not have caught:
   - `build-scout-for-lol` — Prisma 7's `prisma generate` requires `url` in `datasource db`; both schema files are missing it. Compounded by `.dagger/src/image.ts` still pinning `prisma@6` instead of using the package-local `prisma@7.8.0`.
   - `smoke-birmel` — birmel's `start` script runs `prisma db push && bun run src/index.ts` but never `prisma generate`. Worked under Prisma 6 (where `db push` regenerated the client as a side effect); breaks under Prisma 7.
   - `deploy-scout-frontend` — `react: 19.2.6` (pinned exact in scout's frontend) vs `react-dom: ^19.1.1` resolves to `19.2.5`; React refuses to render server-side with mismatched majors-minors.
   - `lint-temporal` — `embedding.ts:214` `let fallbackReason: string | null = null;` trips `no-useless-assignment`. The PR's lint job ran but used a stale Dagger content snapshot of `eslint-config` (`xxh3:4d9893e4ffb90000` on PR vs `xxh3:0e6ee526199ec36a` on main). Same source, different cached deps → different result.

2. **Structural CI gap:** `scripts/ci/src/steps/images.ts:18` defines `MAIN_ONLY = "build.branch == pipeline.default_branch"` and applies it to `build-${pkg}`, `smoke-${pkg}`, `push-${pkg}`, plus `deploy-${site}` in `sites.ts`. PRs emit these steps but Buildkite skips them (state `broken`) and the rolled-up `buildkite/monorepo/pr` check still reports SUCCESS. So Prisma schema validation, image runtime startup, and SSR build skew never ran pre-merge. The same gate also hides `cdk8s-synth` and `clauderon-build` from PRs — both are pure validation with no production side effect.

This plan unbreaks main first (Part A) then closes the gap so the same class of failure can't slip through again (Part B).

## Part B — Close the CI gap

Production-side-effect steps (push to GHCR, push to ChartMuseum, deploy to ArgoCD/cluster, NPM publish, GitHub release, Tofu apply) stay main-only. Site deploys run with `--dryrun` on PRs (build the assets, skip the SeaweedFS sync).

### What stays main-only (intentional)

`push-${pkg}`, `helm-push-${chart}`, `deploy-argocd`, `argocd-health`, `release-please`, `cooklang-push`, `cooklang-release-create`, `npm-*`, `push-ci-base`, `ci-base-version-commit-back`, `tofu-cloudflare`, `tofu-github`, `tofu-seaweedfs`, `version-commit-back`, `clauderon-upload`, `build-summary`. Anything that talks to a production system or publishes a versioned artifact stays gated.

### Audit conclusions

- **Pattern A (Dagger version-pin shadow)** — only `prisma@6` in `.dagger/src/image.ts:646`. No other tool versions are pinned in `.dagger/src/`, `scripts/ci/src/`, or `.buildkite/scripts/`.
- **Pattern B (script depends on tool-upgrade side effect)** — only birmel's `start`. Other DB-using packages either run `prisma generate` explicitly in their script chains or have their image build do it.
- **Pattern C (paired-dep version skew within a workspace)** — only `react`/`react-dom` in scout-for-lol's `frontend` AND `desktop`. No skew in `@trpc/server`/`client`, `@types/react`/`@types/react-dom`, `@aws-sdk/*`, `@opentelemetry/*`, or `prisma`/`@prisma/client`.
- **Pattern D (validation gates hidden from PRs)** — six total: 2 in images, 2 in sites, 1 cdk8s-synth, 2 clauderon-build (per target). All other `MAIN_ONLY` uses are correctly gated to production side effects.
