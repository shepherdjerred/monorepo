# `scripts/setup.ts` cost profiling + helm-codegen removal

## Status

Complete — helm-codegen removal + Deps-phase reliability fixes done & verified. Only the `sjer.red --force` gating was intentionally deferred (risky).

## Context

`bun run scripts/setup.ts` is required before any build/test in a fresh git worktree and was reported as very slow, worst on the first run in a brand-new worktree. Goal: profile where the time goes and cut it.

## Profiling results

Cold first run (worktree off `a3c670b48`, warm `~/.bun` cache, cold `node_modules`):

| Phase                        | Time       | Notes                                                        |
| ---------------------------- | ---------- | ------------------------------------------------------------ |
| Tools                        | 0.2s       | mise trust + install                                         |
| Dependencies                 | 33.2s      | root install 0.1s; 33-pkg fan-out ~30s; homelab subpkgs 3.4s |
| **Build & Generate**         | **119.1s** | **helm-types codegen = 118.4s**; everything else ≤3.5s       |
| Refresh (sjer.red `--force`) | 10.4s      | unconditional `bun install --force`                          |
| Verify                       | 0.0s       | —                                                            |
| **Total**                    | **162.9s** |                                                              |

A warm no-op re-run was _not_ cheaper (288.8s) — nothing is cached/guarded, so `helm-types codegen` (162.5s) and `sjer.red --force` (83.4s) re-ran at full, network-driven cost.

**helm-types codegen was ~73% of the run.**

## Root cause: setup regenerated committed types over the network

- The generated types in `packages/homelab/src/cdk8s/generated/helm/` (23 `*.types.ts`) are **committed** and are the source of truth `homelab` imports.
- `generate-helm-types.ts` nonetheless `rm -rf`'d the dir and **refetched every chart's values/schema over the network** (`helm pull` per chart) on every setup → the 118–162s, plus drift.
- A weekly Temporal schedule already owns regeneration: `helm-types-weekly-refresh` → `runHelmTypesRefresh` (`packages/temporal/src/schedules/register-schedules.ts:187`, activity `packages/temporal/src/activities/helm-types-refresh.ts`) regenerates from live charts and opens a `chore/helm-types-refresh-*` PR if they drifted.

## Fix applied (this branch: `feature/setup-skip-helm-gen`, off `origin/main`)

- **Removed the `helm-types-gen` DAG task** from `scripts/setup.ts` (replaced with a comment pointing to the weekly Temporal refresh). Kept `helm-types-build` (instant; produces `dist/cli.js` checked by Verify).
- Updated `AGENTS.md` phase-4 description (it wrongly listed "helm-types codegen" and "HA types"; HA types aren't generated by setup at all).

### Verification

Cold run after the fix (worktree off `origin/main`):

- **Build & Generate: 119s → 6.0s.** helm codegen gone.
- **Verify: 8/8 artifacts present** — committed helm types intact, build unaffected.
- Total was 254.5s only because this worktree is off `origin/main` (+53 commits of dependency bumps) so `Dependencies` did real cold downloads (231.7s) — a one-time cold-cache cost, not structural.

Steady-state estimate after fix (warm cache): Tools ~3s + Deps ~33s + Build&Generate ~6s + sjer.red ~14s ≈ **~56s vs ~163s before** (~65% faster), entirely from removing helm codegen.

## Deps-phase reliability fixes (also applied this branch)

1. **`--frozen-lockfile`** on the root install and the per-package fan-out (matches the AGENTS.md claim + CI; the script previously ran plain `bun install`, which re-resolves). All 34 packages install cleanly frozen → no lockfile drift on `origin/main`.
2. **Retry-with-backoff** in `installOne` (3 attempts, `attempt*250ms` backoff). Fixes the concurrency flakiness — the warm re-run that failed 3/34 (`homelab/src/cdk8s`, `tasknotes-server`, `tasks-for-obsidian`) now passes **34/34** (each installs fine alone in ~4s; the failures were contention on the shared bun cache).
3. **Surface install errors** — `installOne` now keeps the last error and the failure report prints its stderr/stdout/message via a new `shellErrorDetail()` helper (no type assertion; uses `in` + `instanceof Buffer` narrowing).

### Verification (after all fixes)

Warm run, all green: Tools 0.9s, **Deps 74.4s (34/34)**, **Build & Generate 4.7s**, sjer.red refresh 11.7s, Verify **8/8** → **total 91.7s** (was ~163s cold / 288s warm before, and previously _failing_ on the flaky warm re-run).

## Remaining (deferred)

- **`sjer.red` `bun install --force`** still runs unconditionally (10–83s, network-bound). Gating it correctly is risky — the `--force` re-links the freshly-built local file: deps (webring/astro-og) so sjer.red picks up new dists; skipping it could leave stale local deps. Left as-is. A safe future approach: only `--force` when one of the built local-dep dists changed this run.
- Optionally lower Deps `concurrency` (currently 6) if contention recurs even with retry.

## Workflow Friction

- A **hand-created worktree can't run `bun run scripts/setup.ts`**: `bun` is a mise shim (`~/.local/share/mise/shims/bun`), and mise refuses the untrusted worktree `.mise.toml`, so bun won't launch — setup's own Phase-1 `mise trust` never gets a chance. Normally `claude -w` / the SessionStart hook trusts it; a manual `git worktree add` does not. Workaround: `mise trust -y --all` in the new worktree first (0.06s). Worth having setup document/handle this, or note it in the worktree-workflow skill.
