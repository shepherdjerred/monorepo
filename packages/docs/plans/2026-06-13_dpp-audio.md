# discord-plays-pokemon ‚Äî Audio Pipeline (m4a TS port)

## Status

In Progress

## Context

`discord-plays-pokemon` streams a headless pokeemerald wasm to Discord via Go-Live; today the stream is video-only. The Discord side (`discord-video-stream`'s `AudioStream` / `WebRtcWrapper.sendAudioFrame` / Opus ffmpeg pipeline) is fully ready ‚Äî flipping `includeAudio: true` in `game-streamer.ts` is the last step. The blocker is the emulator host: it never feeds the wasm's audio engine, so no PCM is produced.

This plan hand-ports the GBA m4a (Sappy / MusicPlayer 2000) track-command handlers into TypeScript. Picked over swapping emulators (breaks current wasm-globals game-state reader) or forking the upstream wasm build (adds a build pipeline we don't have).

## Key finding ‚Äî corrects an earlier doc + my own earlier plan

`packages/docs/archive/completed/2026-06-06_headless-pokeemerald-stream.md:64` says "the JS port has no sound path." An initial inspection of the wasm's audio EXPORTS made me think we could "just port" upstream's JS impls ‚Äî but verification of `tripplyons/pokeemerald-wasm` `web/app.js` shows upstream stubs **all 47 audio imports** to `() => 0`. The live `pokeemerald.com` site plays silently. There is no upstream JS to port.

What IS true: the PCM mixer (`m4aSoundMain`, `SoundMain`, `SoundMainBTM`) is in the wasm and exported. The mixer calls the `ply_*` track-command imports each tick. We supply real TS implementations of those imports; the mixer reads them and produces correct PCM. So the heavy lift is the 33 `ply_*` track interpreters + `FadeOutBody` + `TrackStop` + `SampleFreqSet` + `TrkVolPitSet` + `SetPokemonCry*` + `IsPokemonCryPlaying`. The mixer itself (~5 KLOC) is already done in C-compiled-to-wasm.

## Architecture

New `audio/` subdirectory inside the emulator:

```
packages/discord-plays-pokemon/packages/backend/src/emulator/audio/
  index.ts              ‚Äî installAudioImports(env, memory, exports) + createAudioDriver(exports)
  m4a-structs.ts        ‚Äî byte-offset tables + typed accessors
  m4a-state.ts          ‚Äî RAM views + cadence counters
  m4a-handlers-basic.ts ‚Äî simple param setters + SetPokemonCry*
  m4a-handlers-flow.ts  ‚Äî ply_fine/goto/patt/pend/rept, TrackStop
  m4a-handlers-env.ts   ‚Äî LFO, modulation, ADSR, FadeOutBody, ply_port
  m4a-handlers-note.ts  ‚Äî ply_note, ply_xleng, ply_xswee, instrument
  m4a-handlers-ext.ts   ‚Äî ply_xxx dispatch + extended ops
  m4a-driver.ts         ‚Äî per-frame m4aSoundMain call + PCM drain
```

`installAudioImports` runs FIRST in the import loop (overrides `bios.ts`'s catch-all `dispatch ‚Üí 0`). Memory views refreshed via the same `refresh()` pattern as `bios.ts:192-196`. Export refs (`m4aSoundMain` function, `SoundMainRAM_Buffer`/`gSoundInfo` globals) pulled through the runtime-validation pattern from `emulator.ts:30-52` ‚Äî **no `as` casts**.

`emulator.ts` `tick()` gets one new call after `runFrame()`: `audioDriver.tickAndDrain()` calls `m4aSoundMain` (if `WasmRunFrame` doesn't already), reads `gSoundInfo.pcmBuffer` + DMA counter, hands the s8 PCM slice to an `onAudio` callback (mirrors the existing `onFrame` pattern at `emulator.ts:124-126`).

`game-streamer.ts:177` flips `includeAudio: true`; PCM piped into the ffmpeg invocation alongside video; `bitrateAudio: 96`.

### Phase 4 ‚Äî Feed ffmpeg

Pipe PCM into the existing ffmpeg invocation. Flip `includeAudio: true` at `game-streamer.ts:177`; set `bitrateAudio: 96`. Confirm `WebRtcWrapper.sendAudioFrame` is called for each Opus frame.

### Phase 5 ‚Äî e2e audio harness (required gate, local, no Discord)

Three proofs of **real Pokemon audio**:

**Part A ‚Äî manual listen.** Boot to title BGM, capture ~5s, write `out/audio-e2e-source.wav`, run through prod ffmpeg pipeline (capture intermediate `.opus` + round-trip `.wav`), `afplay` source + round-trip.

**Part B ‚Äî mel-spectrogram fingerprint match (the strongest gate).** Committed baseline at `‚Ä¶/src/__tests__/fixtures/title-bgm-baseline.wav` (~5s, captured once via Part A, committed). Test renders fresh PCM and compares:

- Mel cosine similarity (frame-mean) > 0.85 ‚Äî catches silence, noise, wrong rate, wrong song.
- Chroma cosine similarity > 0.90 ‚Äî extra defense against pitch-bend bugs.
- Onset count within ¬±15 % of baseline ‚Äî catches wrong tempo / track-handler regressions.

Implementation: small in-tree DFT, 64 mel bins, 25 ms window @ 10 ms hop, ~150 LOC, no new dep. Baseline regenerates only via `bun run audio-e2e --update-baseline` after Part A approval; CI never auto-rewrites.

**Part C ‚Äî pipeline transport + cheap floors.** `bun test`: feed captured Opus into the same `LibavDemuxer` + `AudioStream` chain `prepareStream` uses internally, stub `WebRtcWrapper.sendAudioFrame`, assert frame count / sizes / cumulative `frametime`. Plus PCM floor checks (RMS, std-dev, FFT band-energy 200 Hz‚Äì8 kHz > 50 %) as cheap sanity.

Part A runs manually; B + C run as `bun test` locally and in Dagger CI.

## Risks

1. **Handler-state divergence from real GBA.** Most likely failure mode. Mitigation: per-handler unit tests + Phase 5 B mel/chroma gates catch gross divergence; diff against agbplay reference if drift appears.
2. **`WasmRunFrame` may already invoke `m4aSoundMain` internally.** If so, calling again would double-tick. Spike in Phase 3: snapshot `gSoundInfo` DMA counter pre/post `runFrame()`.
3. **Sample rate mismatch.** `gSoundInfo.freq` is table-indexed (Emerald ~13379 Hz). ffmpeg resamples to 48 kHz; misreading ‚Üí chipmunk/slow.
4. **Effort overrun.** Few focused days ‚Üí ~2 weeks. If 4+ days without audible BGM, stop and reconsider mGBA-wasm swap.
5. **Licensing.** Don't copy from agbplay verbatim. Write from first principles using `m4a_internal.h` only.
