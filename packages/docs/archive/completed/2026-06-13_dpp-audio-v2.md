# discord-plays-pokemon — Audio v2 (ottohg fork)

## Context

The previous plan (`2026-06-13_dpp-audio.md`) tried to land audio by porting m4a track-command handlers to TS. After shipping all 47 handlers + the per-frame driver + harness in PR #1181, the captured PCM was still silent. Source-level investigation showed why: **the upstream `tripplyons/pokeemerald-wasm` intentionally `#if WASM`-stubs the entire audio engine** — `m4aSoundMain`, `MPlayMain`, `SoundMain`, `MidiKeyToFreq` are all empty function bodies. No mixer exists in the wasm to call our handlers.

The research at `packages/docs/guides/2026-06-13_dpp-audio-research.md` found that someone else already fixed this: [`ottohg/pokeemerald-wasm`](https://github.com/ottohg/pokeemerald-wasm) is an active fork that added `src/m4a_wasm.c` — a full in-tree C reimplementation of the asm-only m4a engine (MPlayMain, every `ply_*`, the DirectSound mixer, BDPCM decode, all CGB channels) plus `--export=gSoundInfo` so the mixed PCM is reachable from JS. Their PCM buffer layout is byte-for-byte what our existing driver already reads.

So the plan flips: **use ottohg's wasm, delete most of PR #1181's host-side handler stack, keep the driver + drain + harness + analysis tooling.**

## Approach (one winner only)

| Decision                                                                                | Why                                                                                        |
| --------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------ |
| Pin ottohg's wasm at a specific SHA                                                     | Their fork is 6 days old and self-described "in-progress"; we don't want surprise breakage |
| Build the wasm in CI via Dagger (not commit pre-built)                                  | Build is deterministic + small enough; matches our existing wasm-fetch pattern             |
| Delete the host-side `ply_*` / struct / memory port                                     | Wasm owns this logic now; running it twice produces wrong audio                            |
| Keep `audio/m4a-driver.ts`, `wav.ts`, `analysis.ts`, `scripts/audio-e2e.ts`             | All pipeline-side concerns that survive the swap                                           |
| Start with s8 PCM drain (existing); upgrade to Float32 `gWasmPcmL/R` if hiss is audible | Lower risk: prove path with the buffer our code already reads                              |

## Phases

### Phase 1 — Build env + reference build

1. Local: `brew install llvm libpng libz` + `uv` already present. Confirm `clang` and `wasm-ld` versions.
2. Clone ottohg at HEAD; record the SHA used.
3. `make wasm` against ottohg at pinned SHA. Time the build. Verify output exports include `gSoundInfo`, `gWasmPcmL`, `gWasmPcmR`, `gSaveBlock1Ptr`, `gPlayerAvatar`, `gObjectEvents`, `VarGet`.
4. Sanity-check the produced wasm passes our existing `emulator-symbols.integration.test.ts` (game-state reader still works).

### Phase 2 — Repoint our vendor + delete the handler stack

1. Update `scripts/fetch-wasm.ts` to fetch from a pinned-SHA built artifact (option A: GitHub release URL; option B: build locally via `scripts/build-wasm.sh` and check in; option C: Dagger task in CI builds + stamps sha256).
2. Update the committed `pokeemerald.wasm.sha256`.
3. Delete the handler files + their tests.
4. Strip `audio/index.ts` to just the driver factory.
5. Strip `bios.ts` back to its pre-PR-1181 shape (no `extras` override).
6. Strip `emulator.ts` back: drop `initAudio()`, keep `onAudio()` + per-frame drain call.
7. `bun run typecheck` + `bun test` + `bunx eslint` all clean.

### Phase 3 — Drain verification

1. `bun run scripts/audio-e2e.ts --no-play --frames 600` (~10 s of game time). RMS should be **non-zero**.
2. If PCM is non-zero but garbled, check sample rate (`gSoundInfo.pcmFreq` should be ~13379 Hz) and buffer offset (alternate halves via `pcmDmaCounter`).
3. If RMS is still 0, file an issue upstream; the symptom should point at a specific missing export or wrong base address.

### Phase 4 — Listen + baseline + fingerprint gate

1. `bun run scripts/audio-e2e.ts` (with `afplay`). Confirm the title-screen BGM is audibly Pokemon Emerald's theme.
2. `bun run scripts/audio-e2e.ts --update-baseline` after the listen confirms it's right.
3. Write `audio-fingerprint.test.ts`: boot wasm, capture ~5 s of title BGM, mel cosine > 0.85 + chroma cosine > 0.90 + onset count within ±15 % of baseline.
4. Both the test and the baseline WAV get committed.

### Phase 5 — Wire to game-streamer

1. Flip `includeAudio: true` at `packages/discord-plays-pokemon/packages/backend/src/stream/game-streamer.ts:177`.
2. Wire the `onAudio` callback to write PCM into the ffmpeg pipe (separate stdin or muxed).
3. Set `bitrateAudio: 96`.
4. Confirm `WebRtcWrapper.sendAudioFrame` is called for each Opus frame produced (the existing pipeline-transport test still passes).

### Phase 6 — Docs + close-out

1. Strike "the JS port has no sound path" in `packages/docs/archive/completed/2026-06-06_headless-pokeemerald-stream.md:64`.
2. Mark this plan `Status: Complete`; `git mv` to `packages/docs/archive/completed/`.
3. Move the old `2026-06-13_dpp-audio.md` to `packages/docs/archive/superseded/`.
4. Delete `packages/docs/todos/dpp-audio-handlers.md` (resolved by the wasm swap).
5. Update PR #1181 description (or open a fresh PR if we want a cleaner diff against `main`).

## Verification

- **Layer 1** — `bun run scripts/audio-e2e.ts --no-play --frames 600`: RMS > 5 on s8 scale, PCM has spectral content > 200 Hz.
- **Layer 2** — `bun run scripts/audio-e2e.ts`: you listen; confirms it's the Pokémon Emerald theme. **You are the gate.**
- **Layer 3** — `bun test audio-fingerprint`: mel + chroma + onset match the committed baseline.
- **Layer 4** — `bun run typecheck` + `bun test` (full backend suite, currently 100/100).
- **Layer 5** — Manual Discord smoke: bot Go-Lives, BGM audible at ≥5 min, no desync.

## Risks

| Risk                                                                | Likelihood              | Mitigation                                                                                                                                         |
| ------------------------------------------------------------------- | ----------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------- |
| ottohg fork has a bug for some specific song/SFX/cry                | High (in-progress code) | Pin SHA; file upstream issues; carry local patch in `scripts/build-wasm.sh` if a quick fix found                                                   |
| Build env breaks in CI (uncommon toolchain)                         | Medium                  | Dagger image; cache the built wasm artifact across runs                                                                                            |
| Game-state reader breaks (some `--export-all`'d global got dropped) | Low                     | `emulator-symbols.integration.test.ts` catches this; verify upfront in Phase 1 step 4                                                              |
| Output wasm grows too large to comfortably ship                     | Low                     | ~+6–10 MB expected (song table + voicegroups + samples). Manageable; can sub-set songs if needed via `WASM_SONG_NAMES` Makefile var ottohg exposes |
| Float32 drain (optional upgrade) destabilises pipeline              | Low                     | Only attempt after the s8 path proves audible; revert if any wobble                                                                                |
