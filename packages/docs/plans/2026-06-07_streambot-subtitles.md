# Streambot — Subtitle Support

## Status

Complete

## Context

`streambot` streams a single H.264 video into a Discord voice channel via the in-repo fork
`@shepherdjerred/discord-video-stream`. A Go-Live stream carries one video track (no separate caption
channel), so subtitles must be **burned into the video** with ffmpeg's `subtitles=` (libass) filter.
This adds subtitle support for the two requested cases: **YouTube/URL** (yt-dlp) and **local files**
(sidecar + embedded). Based off `main` (which already vendored the fork + added `/stream seek`).

Validated against the live `media-streambot` pod and real `/media/movies` + `/media/tv` libraries:
sidecars use Plex/Bazarr naming (`<base>.<lang>[.forced|.hi].srt`), embedded Remux subs are mostly PGS
(image, un-burnable → covered by sidecar), and the burn pipeline (`scale,subtitles=`) works in the
production image (libass + fonts present).

## Implementation

| Area            | Change                                                                                                                                                                                                                                                                                       |
| --------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Fork filter API | `videoFilters?: string[]` added to `prepareStream` (`packages/discord-video-stream/src/media/newApi.ts`), applied after `scale` via the pure `buildVideoFilterChain`. Re-applied on every seek/HW→SW restart by the seekable player.                                                         |
| Config          | `subtitles` block (`enabled`, `languages`, `includeAutoGenerated`) + `ffprobePath` (`src/config/schema.ts` + env in `config/index.ts`: `SUBTITLES_ENABLED`, `SUBTITLE_LANGUAGES`, `SUBTITLES_INCLUDE_AUTO_GENERATED`, `FFPROBE_PATH`).                                                       |
| Source pref     | `SubtitlePrefSchema` (optional `enabled`/`language`) on each `Source` variant (`src/sources/source.ts`).                                                                                                                                                                                     |
| Resolution      | Pure helpers in `src/sources/subtitles.ts`; ffprobe/ffmpeg/yt-dlp glue in `src/sources/subtitle-io.ts`. Sidecar-first, then embedded **text** extraction (PGS/VobSub/DVB skipped), then yt-dlp download (human → auto-caption). Every track staged to `$TMPDIR/streambot-subs/<uuid>.<ext>`. |
| Wiring          | `ResolvedSource.subtitle?` (`machine/types.ts`); `resolve.ts`/`ytdlp.ts` attach it; `streamer.ts` forces software encode when subs present, passes `videoFilters`, and unlinks the temp file in `runStream`'s `finally`; `index.ts` sweeps orphans at startup.                               |
| Command         | `subtitles` (on/off) + `sublang` options on `/stream play` and `/stream playnext`, threaded into the source (incl. playlist items); ack shows the override.                                                                                                                                  |

## Testing (layered, no silent skips)

- **Unit** (`test/subtitles.test.ts`, +`command-handler.test.ts`): pure helpers (name parse, ranking,
  codec classify, escaping, effective-config, yt-dlp args, embedded pick using the real Dune ffprobe
  fixture) + command option threading. **117 streambot unit tests pass.**
- **Fork no-spawn seams** (`discord-video-stream/test/newApi.filters.test.ts`, `player.test.ts`):
  `buildVideoFilterChain` ordering; `videoFilters` reaches `prepareStream` on start **and after seek**.
  **13 fork tests pass.**
- **Real-ffmpeg integration** (`integration/subtitles.integration.test.ts`, `bun run test:integration`):
  sidecar staging, embedded text extraction (asserts cue text), and the actual `scale,subtitles=` burn
  (frame-diff). Runs only in the streambot image via the `smoke-test-streambot` Dagger fn (the plain
  `bun test` is scoped to `test/` and has no ffmpeg). Hard-fails if a binary is missing.
- **e2e** (`e2e/run.ts`): generates a sidecar `.srt` and plays with subtitles on (manual `e2eStreambot`).

## Key decisions / notes

- **Software encode when subs on** — libass is a CPU filter incompatible with the VAAPI hardware-frame
  graph; VAAPI still used for subtitle-free videos. VAAPI burn-in (hwdownload→subtitles→hwupload) and
  image-sub (PGS/VobSub) burn-in are deferred.
- **Fork consumed via dist .d.ts in CI** (`BUILD_TIME_DEPS` builds it) so streambot's strict `tsc`
  resolves the fork's types from `dist` + `skipLibCheck`. Locally, the dev symlink makes `bun run
typecheck` in streambot read the fork's _source_ and report ~46 pre-existing fork-internal errors
  (48 on plain `main`); streambot's _own_ code is type-clean and the fork's own typecheck passes.

### Caveats

- Software-only encode while subtitles are burned (perf trade-off vs reliability).
- Image subtitles (PGS/VobSub/DVB) are skipped, not burned — rely on a text sidecar.
- yt-dlp subtitle resolution is best-effort (absence → no subtitles, not an error).
- Local `bun run typecheck` in streambot shows pre-existing fork-internal errors via the dev symlink;
  CI is green because it builds the fork's `dist` first. Don't "fix" those fork errors here.
